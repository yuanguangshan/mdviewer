## 一、stream：先要明确，而不是猜（v1.11.24）

### 1.1 一个被误会的「写死」

用户问：「ai 接口请求的时候你是不是写死了模式：stream？」

我先去代码里 grep 了一遍 `stream`、`callAiApi`、`fetch(`。结论很干脆：**没有写死，甚至恰恰相反——完全没用流式。**

那时 `callAiApi` 发出的请求体只有三个字段：

```js
body: JSON.stringify({
  model,
  messages: [
    { role: 'system', content: systemPrompt },
    { role: 'user',   content: promptText }
  ],
  temperature: 0.7
})
```

没有 `stream` 键。而 OpenAI 兼容接口在 `stream` 缺省时默认 `false`，所以走的是**非流式**：等全部返回后直接 `resp.json()`，读 `choices[0].message.content` 整段文本。

这本来没毛病。但用户说了一句关键的话：**「显式指定 stream:false 吧，我的接口有可能默认用流式的。」**

这句话里有经验之谈。用户的接口（自建代理 `wx.want.biz`）行为不可控，万一默认流式，而前端又没声明，返回的就是 SSE 流而非 JSON，到时候 `json()` 必然炸。**与其赌接口默认值，不如客户端主动把行为钉死。** 这是「防御性明确」的好例子——不依赖对端的隐式约定。

于是我在请求体里加上了 `stream: false`：

```js
body: JSON.stringify({
  model,
  messages: [
    { role: 'system', content: systemPrompt },
    { role: 'user',   content: promptText }
  ],
  temperature: 0.7,
  stream: false
})
```

只动了一个字段，没升版本号——理由也站得住：这是「行为明确化」，不是新功能。

### 1.2 但故事没完

升版本、提交、推送都顺理成章。可我们都没想到，这一个 `stream: false`，正是下一幕爆炸的引信。

## 二、服务端不守规矩：SSE 与 JSON 的兼容战（v1.11.25）

### 2.1 报错来得很快

用户把控制台红字甩了过来：

```
app.js:875 SyntaxError: Unexpected token 'd', "data: {"ch"... is not valid JSON
callAiApi @ app.js:875
await in callAiApi
aiProcessSelection @ app.js:875
aiPolish @ app.js:912
```

`Unexpected token 'd'`、`"data: {"ch"...` —— 这形状太眼熟了。`data:` 开头，明显是 **SSE（Server-Sent Events）流**的帧格式。也就是说，服务端**压根没理我们的 `stream: false`**，照样返回了流式响应，而前端还在用 `resp.json()` 去解析整段流，自然报「这不是合法 JSON」。

根因定位：用户的自建代理（`wx.want.biz`）在收到 `stream:false` 时忽略了它，仍然以 SSE 方式把 `data: {...}` 一行行吐出来。这不是我们前端能改的服务端行为。

### 2.2 最稳的修法：解析层同时兼容两种形态

我当时的判断是：与其纠结服务端为什么不守规矩，不如让**响应解析同时兼容流式和非流式**。这样无论服务端听不听话都能拿到内容。

新增 `extractAiContent(text)`：

```js
function extractAiContent(text) {
  const trimmed = (text || '').trim();
  if (trimmed.startsWith('data:')) {
    // 流式：按 SSE 逐行抽 delta.content
    let out = '';
    for (const rawLine of trimmed.split('\n')) {
      const line = rawLine.trim();
      if (!line.startsWith('data:')) continue;
      const payload = line.slice(5).trim();
      if (!payload || payload === '[DONE]') continue;
      try {
        const j = JSON.parse(payload);
        const ch = j.choices && j.choices[0];
        if (ch && ch.delta && ch.delta.content) out += ch.delta.content;
        else if (ch && ch.message && ch.message.content) out += ch.message.content;
      } catch (_) { /* 忽略 keepalive / 非 JSON 行 */ }
    }
    return out.trim();
  }
  // 非流式 JSON
  try {
    const data = JSON.parse(trimmed);
    return (data.choices?.[0]?.message?.content || '').trim();
  } catch (e) {
    throw new Error('返回既非 SSE 流也非合法 JSON：' + trimmed.slice(0, 80));
  }
}
```

关键细节：
- 以原始文本是否 `startsWith('data:')` 来区分流式/非流式，因为 SSE 和 JSON 的「开头字节」天然不同。
- 流式分支要宽容：跳过空行、`[DONE]` 结束标记、以及任何解析失败的行（keepalive 心跳往往是 `: ping` 之类的注释行）。
- 保留 `stream:false` 请求——万一服务端哪天「改邪归正」返回干净 JSON，我们就优雅走下方分支。

`callAiApi` 改为 `await resp.text()` 再交给它解析。

### 2.3 单测兜底

我写了个 `extract_test.mjs`，用 node 直接跑三种输入：
1. 模拟用户的 SSE 流 → 期望拼出 `你好，世界`；
2. 标准非流式 JSON → 期望 `plain json`；
3. 完全非法的纯文本 → 期望抛错。

三场景全绿。提交 v1.11.25，推送。

**这一幕的教训**：永远不要假设对端会按文档履约。把「输入形态的探测与兼容」放在解析层，比在请求层反复讨价还价要稳得多。

## 三、流式打字机的诞生（v1.11.26）

### 3.1 用户想要「打字机」

v1.11.25 让功能跑通了，但体验是「等全部返回再一次性替换选区」。用户问：「流式回复你这边好处理吗？想要打字机效果。」

我说好处理——而且简直是顺水推舟。因为服务端**本来就在流式返回**，只是我们之前把流攒完才显示。只要改成「边读 `ReadableStream` 边抽 `delta.content` 回填」，真·打字机就成了。

### 3.2 核心改造

新增 `streamAiApi()`，替代原来的整段等待：

```js
async function streamAiApi(promptText, systemPrompt, onToken, signal) {
  // ... 读取配置、鉴权 ...
  const resp = await fetch(endpoint, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json', 'Authorization': 'Bearer ' + apiKey },
    body: JSON.stringify({ model, messages, temperature: 0.7, stream: true }),
    signal            // 支持 AbortController 中途停止
  });
  if (!resp.ok) { /* 错误处理 */ }
  const reader = resp.body.getReader();
  const decoder = new TextDecoder();
  let buffer = '', sseMode = null, full = '';
  while (true) {
    const { value, done } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });
    // 首块探测：data: 开头是 SSE，{ 开头是 JSON 兜底
    if (sseMode === null) {
      const head = buffer.trimStart();
      if (head.startsWith('data:')) sseMode = true;
      else if (head.startsWith('{')) sseMode = false;
    }
    if (sseMode === true) {
      let idx;
      while ((idx = buffer.indexOf('\n')) >= 0) {
        const line = buffer.slice(0, idx).trim();
        buffer = buffer.slice(idx + 1);
        if (!line.startsWith('data:')) continue;
        const payload = line.slice(5).trim();
        if (!payload || payload === '[DONE]') continue;
        try {
          const j = JSON.parse(payload);
          const ch = j.choices && j.choices[0];
          const delta = (ch && ((ch.delta && ch.delta.content) || (ch.message && ch.message.content))) || '';
          if (delta) { full += delta; if (onToken) onToken(delta, full); }
        } catch (_) {}
      }
    }
  }
  if (sseMode === false) {
    const data = JSON.parse(buffer.trim());
    full = (data.choices?.[0]?.message?.content || '').trim();
    if (onToken) onToken(full, full);
  }
  return full.trim();
}
```

最棘手的部分是 **SSE 跨块边界**：网络分片未必按 `\n` 对齐，`data: {...}` 一行可能被切成两半。所以我用 `buffer` 累积、只在遇到完整 `\n` 时才处理一行，残留部分留给下一轮 `read()`。这是流式解析最容易翻车的地方，我专门写了测试：把一段 SSE 流**每 3 字节切一刀**，模拟极端分片，确认跨块不断裂、token 顺序不乱。测试通过。

### 3.3 一个浮层：实时显示 + 三个按钮

光解析不够，得有「看得见的打字机」。新增浮层 `#aiStreamPanel`：

- 实时把 `onToken` 收到的增量文本填进去，带闪烁光标；
- **应用**：把全文写回编辑器（润色/扩写/翻译 → 替换选区；总结 → 插文首；生成 → 插光标处）；
- **停止**：`AbortController.abort()`，中止生成但保留已得内容，仍可应用（不报红）；
- **取消**：丢弃关闭。

五个 AI 动作（润色/扩写/翻译/总结/生成）全部改走 `streamAiApi`。

### 3.4 验证

`node --check` 通过；单测覆盖「SSE 每 3 字节切分跨块」「JSON 兜底」「空响应」三场景全绿。提交 v1.11.26 推送。

后来用户实测，用 gemini 模型真生成出三段带 `###` 标题的长文——打字机效果端到端跑通了。那一刻我们知道，AI 这块从「能用」跨到了「好用」。

## 四、两个收尾小项（v1.11.27）

### 4.1 AI「测试连接」按钮

用户说：「处理一下，加上测试连接，音乐自动搜索。」

先说测试连接。我加了 `testAiConnection()`，关键设计是——**优先用表单当前填的值，没点保存也能测**：

```js
async function testAiConnection() {
  const ep = $('#aiEndpoint').value.trim() || 'https://api.openai.com/v1/chat/completions';
  const md = $('#aiModel').value.trim() || 'gpt-4o-mini';
  const key = $('#aiKey').value.trim();
  if (!key) { toast('请先填写 API Key 再测试', 'err'); return; }
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), 20000); // 20s 超时
  try {
    const resp = await fetch(ep, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json', 'Authorization': 'Bearer ' + key },
      body: JSON.stringify({ model: md, messages: [{ role: 'user', content: 'ping' }], max_tokens: 5, stream: false }),
      signal: controller.signal
    });
    if (resp.ok) {
      let model = md;
      try { model = (await resp.json()).model || md; } catch (_) {}
      toast('连接成功 ✓（' + model + '）', 'ok');
    } else { /* 展示 HTTP 错误 / CORS 提示 */ }
  } catch (e) { toast('连接失败：' + aiErrorHint(e, ep), 'err'); }
  finally { clearTimeout(timer); }
}
```

发一个 `max_tokens:5` 的极小请求，既验证端点存活、Key 有效，又不会消耗什么 token，更不会往正文插任何东西。按钮放在弹窗的「取消 / 测试连接 / 保存」之间。

### 4.2 音乐自动搜索：关掉「自动」

音乐的报错有来头。用户之前贴过控制台里 `iTunes` 相关的失败日志。我第一反应是「是不是代码里写了自动搜 iTunes」——结果 `git log -S "itunes"` 一查，**git 历史里根本没有 iTunes 字样**。

真相是：音乐面板是个 `iframe`，`src` 写死在页面加载时就指向外链 `wealth.want.biz/music`。**那个页面自己会在加载时自动搜 iTunes**（用户看到的报错来自外链），mdviewer 控制不了外链页面的内部逻辑。

我能做的是**关掉「自动」**：把 `src` 改成 `data-src`，只有**首次点 🎵 展开时才真正加载**。

```html
<iframe class="lib-music-frame" data-src="https://wealth.want.biz/music/index.html" ...></iframe>
```

```js
libMusicToggle.addEventListener('click', () => {
  const open = libMusic.hidden;
  libMusic.hidden = !open;
  if (open && !libMusic.dataset.loaded) {
    const frame = libMusic.querySelector('iframe');
    frame.src = frame.dataset.src;   // 首次展开才注入，关闭自动搜索
    libMusic.dataset.loaded = '1';   // 之后不再重载，保留播放状态
  }
  // ...
});
```

效果：一开页面不再触发外链搜索和那些报错；想听歌时点一下照常可用，且重复展开不会重载（保护播放状态）。这是「不动外链代码、只改触发时机」的务实解法。

提交 v1.11.27 推送。

## 五、用户给的公式考题：KaTeX 接入（v1.11.28）

### 5.1 「好像不行」

用户甩来一大段含 `\oiint`（闭合曲面积分）的高斯公式，说「好像不行」，并附了若干变体写法。这显然是在考我刚集成的 KaTeX。

我的第一反应是排查两个可能：① KaTeX 是否支持 `\oiint`；② 我是不是漏加载了 CSS。

先实测。在 node 里用 Function 注入真实 `katex.min.js`，把用户给的 9 个公式全跑一遍 `renderToString`：

```
✅ OK    高斯1(oiint)
✅ OK    高斯2(oiint)
✅ OK    微分形式
✅ OK    梯度形式(oiint)
✅ OK    全分量(oiint)
✅ OK    测试小贴士(oiint)
✅ OK    仅oiint
✅ OK    iiint对比
✅ OK    oint对比
```

**9/9 全过**。结论很明确：KaTeX 0.16 原生就支持 `\oiint`，连宏都不用加——用户那句「KaTeX 原生支持 \oiint，不需要 esint 包」判断完全正确，是我多虑了。

### 5.2 那为什么用户觉得「不行」

我用 jsdom 加载真实 KaTeX + auto-render，测了三种场景：

| 场景 | 结果 |
|---|---|
| 正文 `<p>$$…$$</p>` | ✅ 渲染成 `.katex`（含 `\oiint`） |
| 代码块 ` ``` ` 内 | ⚠️ **按设计不渲染**，显示 `$$` 源码 |
| 行内 `$…$` | ✅ 渲染 |

真相浮现：**用户是把他那堆公式放在消息里的代码块（` ``` ` 包裹）给我的**。代码块是「源码展示区」，KaTeX 默认跳过 `pre`/`code`（我的 `ignoredTags` 也这么写）。于是在他眼里「公式没渲染」，其实它被正确当源码展示了。另一个可能是没硬刷新、跑着旧 SW 缓存。

### 5.3 既然要测，那就把 KaTeX 真的接好

用户说「接入 KaTeX」，于是我把它做成了正式能力，延续项目一贯的「全本地 vendor、离线优先」风格：

1. **下载本地资源**（不依赖 CDN，PWA 离线也能渲染）：
   - `vendor/katex.min.js`（277K）
   - `vendor/katex.min.css`（23K）
   - `vendor/katex-auto-render.min.js`（3.5K）
   - `vendor/fonts/` 下 20 个 `woff2` 字体（共 296K，只取 woff2，现代浏览器优先用它）

2. **按需懒加载**（和 mermaid 同策略，不拖首屏）：

```js
let katexLoading = null;
async function ensureKatex() {
  if (window.katex && window.renderMathInElement) return;
  if (!katexLoading) {
    katexLoading = (async () => {
      if (!document.getElementById('katex-css')) injectKatexCss();
      await loadScript('./vendor/katex.min.js');
      await loadScript('./vendor/katex-auto-render.min.js');
    })();
  }
  return katexLoading;
}

function hasMath(text) {
  return /\$\$[\s\S]+?\$\$/.test(text)
    || /(^|[^\\$])\$[^$\n]+\$/.test(text)
    || /\\\(|\\\[/.test(text);
}

async function renderMathFormulas() {
  if (!hasMath(editor.value)) return;   // 没有公式就不加载
  await ensureKatex();
  window.renderMathInElement(preview, {
    delimiters: [
      { left: '$$', right: '$$', display: true },
      { left: '\\[', right: '\\]', display: true },
      { left: '$',  right: '$',  display: false },
      { left: '\\(', right: '\\)', display: false }
    ],
    throwOnError: false,
    ignoredTags: ['script', 'noscript', 'style', 'textarea', 'pre', 'code']
  });
}
```

几个值得记的点：
- `hasMath` 正则要**避免误触发**：价格 `$30`、跨行的两个 `$`、转义 `\$` 都不应触发。我写了 8 个用例单测，全绿。
- `ignoredTags` 包含 `pre`/`code`——这是用户「代码块不渲染」现象的根，也是正确设计：代码块里的 `$$` 应当原样显示。
- 渲染时机放在 DOMPurify 之后，公式不会被 sanitize 剥掉；每次预览刷新幂等重渲。
- SW 对同源 GET 已是缓存优先 + 后台更新，KaTeX 首次用后自动进缓存，只升了 `CACHE_NAME`。

3. **jsdom 真渲染验证**：加载**真实 KaTeX**，把用户那段 `$$\text{古希腊：本体论} \longrightarrow \text{近代：认识论}$$` 喂进去，成功生成 `.katex-display`、`$$` 定界符被正确消费。

我顺手给用户生成了一个 `katex-demo.md`，让他直接粘到正文（不包代码块）验证。提交 v1.11.28 推送。

**这一幕的附带认知**：用户给的 `[gemini-3.5-flash-lite]` 这种标签，是他代理的正常回声，不是我们的锅。用户也确认「我的接口正常行为」，于是我们决定不清洗——这是「尊重对端契约、不越界代劳」的又一个例子。

## 六、「最后一公里」：文库静默增量同步 NAS（v1.11.29）

### 6.1 用户的洞见

用户写道：

> 跨设备同步的「最后一公里」——NAS 上传/下载已经铺好了路，但目前还是手动触发。如果能让「文库」在后台静默检测网络，并与 NAS 做增量同步……它就真的从一个「本地工具」进化为「私人云笔记」了。这里修改了可自动同步到 NAS，NAS 不用同步编辑器，需要时我自己拉。

这段话信息密度极高，我逐字读了几遍，提炼出几个要点：
1. 单向模型：**本地改 → 静默推 NAS，NAS 不主动推回来，需要时手动拉**；
2. 触发要静默：后台检测网络、增量同步，不打扰；
3. 可用 `navigator.storage` 管配额，防文库被浏览器当缓存清掉；
4. 契合 Markdown 编辑器真实用法：修改大概率发生在本地某台设备。

我完全赞同，而且认为这个单向模型是整个方案里**最聪明的一点**——它彻底规避了双向同步最头疼的「同一篇在两设备都改了」的冲突，因为 NAS 永远只是本地镜像。

### 6.2 基于现有代码的可行性确认

动手前我先读了 `writebackLib` 和 NAS 上传代码。文库（libDocs）每篇已经是 `{ id: genId(), name, content, updatedAt: Date.now(), history? }`，而 `writebackLib()` 在编辑后 800ms 去抖回写 IndexedDB。

这**天然是增量同步的燃料**：
- 增量判断：`updatedAt > 上次同步时间` 即可，无需额外算法；
- 复用现有 NAS 接口（`/api/upload` 单文件 multipart + Basic 鉴权），**NAS 服务端零改动**；
- 触发点就挂在 `writebackLib()` 之后。

### 6.3 落地实现

```js
const NAS_SYNC_STATE_KEY = 'nas-sync-state';   // id -> 上次成功同步的 updatedAt
const SYNC_DEBOUNCE_MS = 4000;

function loadSyncState() {
  try { return JSON.parse(localStorage.getItem(NAS_SYNC_STATE_KEY) || '{}'); } catch (_) { return {}; }
}
function saveSyncState(state) {
  try { localStorage.setItem(NAS_SYNC_STATE_KEY, JSON.stringify(state)); } catch (_) {}
}

async function syncLibraryToNas() {
  const auth = localStorage.getItem('nas-auth');
  if (!auth) return;                 // 未配置凭据 → 静默跳过
  if (!navigator.onLine) return;     // 离线 → 静默跳过（留待 online 事件补传）
  const docs = await idbGetAll('lib');
  const state = loadSyncState();
  const pending = docs.filter(d => (d.updatedAt || 0) > (state[d.id] || 0));
  if (!pending.length) return;
  setSyncDot('syncing');
  for (const d of pending) {
    try {
      const blob = new Blob([d.content], { type: 'text/markdown' });
      const fd = new FormData();
      fd.append('file', blob, d.id + '.md');   // 文件名用文库 id，稳定唯一、绝不重名覆盖
      const resp = await fetch(NAS_UPLOAD_URL, {
        method: 'POST',
        headers: { Authorization: 'Basic ' + btoa(auth) },
        body: fd
      });
      if (resp.ok) { state[d.id] = d.updatedAt; saveSyncState(state); }
      else if (resp.status === 401) { setSyncDot('error'); break; }
    } catch (_) { setSyncDot('error'); }
  }
  setSyncDot(pending.length ? 'synced' : 'idle');
}
```

调度（挂在初始化区）：
```js
// 编辑后去抖触发（4s）
const scheduleSync = debounce(syncLibraryToNas, SYNC_DEBOUNCE_MS);
// online 事件立即补传
window.addEventListener('online', syncLibraryToNas);
// 启动补传
syncLibraryToNas();
// 每 3 分钟兜底
setInterval(syncLibraryToNas, 3 * 60 * 1000);
// 请求持久化存储，防文库被浏览器当缓存清掉
if (navigator.storage && navigator.storage.persist) {
  navigator.storage.persist().catch(() => {});
}
```

状态栏小圆点（`#syncDot`，灰/蓝脉冲/绿/红），hover 看详情，**不弹任何 toast**——完全静默。

### 6.4 测试：最硬核的一环

我写了 `sync_test.mjs`，用**字符串感知的括号扫描**从真实 `app.js` 里精确抽取 `loadSyncState`/`saveSyncState`/`setSyncDot`/`syncLibraryToNas` 这几个函数 + 常量声明，注入 mock（`idbGetAll`/`fetch`/`localStorage`/`navigator`/`Blob`/`FormData`），跑 7 个场景：

```
✅ 无凭据不调用 fetch
✅ 离线不调用 fetch
✅ 仅 b 上传（增量：a 的100不>100）
✅ state[b]=200
✅ state[a] 仍为 100
✅ 空 state → a,b 上传(2次)
✅ fetch 失败后 state 不写入(留待补传)
```

调试过程中踩了个坑：第一版提取漏了 `NAS_SYNC_STATE_KEY` 常量，导致 `loadSyncState` 读 `localStorage.getItem(undefined)` 恒返回 `{}`，过滤判定失真——测试失败是**测试提取不全**，不是产品代码 bug。补上常量声明后 7/7 全绿。这提醒我：抽真实代码做集成测试时，依赖的常量、闭包变量必须一并带入，否则测的是「残缺副本」。

提交 v1.11.29 推送。

**这一幕的边界决策**（和用户确认过）：
- 文件名用 `{id}.md`，不含标题——避免在 NAS 上重名相互覆盖。代价是跨设备「手动拉」回编辑器时不恢复标题名，但用户明确不要双向复杂度，所以保持现状。
- 导出带公式暂未做（HTML/PDF 导出用渲染前快照，公式仍是 `$$` 源码）——这是已知局限，记在日志里待将来。

## 七、意料之外的 SW 报错（v1.11.30）

### 7.1 报错

用户贴来：

```
sw.js:72 Uncaught (in promise) TypeError: Failed to execute 'put' on 'Cache':
Request scheme 'chrome-extension' is unsupported
    at sw.js:72:65
```

这条错很隐蔽，只在装了浏览器扩展（密码管理器、翻译、Grammarly 之类）的环境出现。

### 7.2 根因

`sw.js` 的 fetch 拦截器会对页面里**所有 GET 请求**尝试写缓存：

```js
const network = fetch(req).then((fresh) => {
  if (fresh && (fresh.ok || fresh.type === 'opaque')) cache.put(req, fresh.clone());
  return fresh;
}).catch(() => null);
```

而浏览器扩展注入的请求是 `chrome-extension://` scheme。**Cache API 只支持 `http:` / `https:`**，遇到 `chrome-extension://` 的 `cache.put()` 就抛 `Request scheme 'chrome-extension' is unsupported`。

更隐蔽的是：这个 `cache.put` 在 `.then` 里，而外层 `.catch(() => null)` 只捕获 `fetch` 的拒绝，**捕获不到 `cache.put` 本身的拒绝**，于是变成未捕获的 Promise 拒绝，控制台一片红。

### 7.3 修复

两处改动：

1. fetch 拦截开头加 scheme 守卫，非常规 scheme 直接透传、不缓存：

```js
const url = new URL(req.url);
if (url.protocol !== 'http:' && url.protocol !== 'https:') return;
```

2. 两处 `cache.put`（`navigate` 分支 + 静态资源分支）都加 `try/catch` 防御，杜绝未来任何异常响应再变成未捕获拒绝：

```js
const network = fetch(req).then(async (fresh) => {
  if (fresh && (fresh.ok || fresh.type === 'opaque')) {
    try { await cache.put(req, fresh.clone()); } catch (_) {}
  }
  return fresh;
}).catch(() => null);
```

`node --check` 通过。提交 v1.11.30 推送。

**这一幕的教训**：Service Worker 的 fetch 拦截器是「全局放大器」——任何一行对请求/响应的假设（这里是「scheme 都是 http/https」），都会被页面里所有请求放大。写 SW 时对所有外部输入（scheme、type、status）做防御性判断，是基本功。

## 八、一处文案的诚实修正（v1.11.31）

### 8.1 用户指着截图问

用户发了张截图，指向文库底部那行字：「🔒 本地保存 · 不上传」，问「这里是否要改一下」。

我立刻意识到：这行字是 NAS 同步**之前**写的。现在文库已经会**自动增量同步到 NAS** 了，还写「不上传」就是**误导用户**——文档在配置 NAS 后确实会传。

### 8.2 改

```html
<span class="lib-foot-note" title="文档优先保存在本机；配置 NAS 后会自动增量同步">
  🔒 本地优先 · NAS 同步可选
</span>
```

提交 v1.11.31 推送。

我顺口问了用户要不要做成动态的（未配 NAS 显示「不上传」，配了显示「同步中」），用户说「就这样吧」——静态准确文案已足够避免误导。

**这一幕看似最小，却最考验「诚实」**：功能演进后，UI 文案必须同步演进。一句过时的「不上传」，会让用户对隐私边界产生错误预期。及时修正，是对用户信任的尊重。

| v1.11.31 | 文库文案修正 | 诚实修正 |

几条反复验证的原则，值得写进团队记忆：

1. **不假设对端履责**：服务端可能忽略 `stream:false`、可能返回非预期形态。解析层做形态探测与兼容，比在请求层讨价还价稳。
2. **尊重对端契约、不越界代劳**：`[gemini-*]` 标签是用户代理的正常回声，我们不动；代码块不渲染公式是正确设计，不是 bug。
3. **复用优于重构**：文库同步复用现有上传接口，NAS 服务端零改动；KaTeX 走与 mermaid 相同的懒加载策略。
4. **抽真实代码做集成测试**：用字符串感知提取 + mock 注入，能把「增量过滤」「SSE 跨块」「scheme 守卫」这类易错点真正验证。注意把依赖的常量和闭包变量一并带入。
5. **SW 是全局放大器**：任何对请求/响应的假设都会被放大，必须防御性判断 scheme/type/status。
6. **功能演进，文案同步**：过时的「不上传」会误导用户对隐私的预期，及时修正。
7. **单向同步是最聪明的简化**：当冲突处理成本高时，用「单向镜像 + 手动拉」换掉「双向合并」，往往更稳、更省心。

mdviewer 今天从一个「带 AI 的本地 Markdown 编辑器」，实实在在地往「私人云笔记」迈了一大步。而推动它前进的，不是某个宏大规划，恰恰是一次次「这里是不是要改一下」的细碎对话。

这大概就是结对编程最舒服的样子。
