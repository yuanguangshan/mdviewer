# 从一只笔到一座书房 —— mdviewer 项目的协作评述

> 作者按：这是一篇关于我们共同完成的 Markdown 编辑器项目（mdviewer）的评述。它不是产品说明书，也不是技术白皮书，而是一份写给"我们一起做过的事"的复盘与致意。如果你是从零开始读到这里的读者，它也能当作一份完整的产品演化史料。

---

## 一、缘起：一只足够好的笔

一切的开始其实很朴素。我们要的不是一个臃肿的写作软件，而是一支"笔"——打开网页就能写 Markdown，左边写、右边看，干净、即时、不打扰。没有账户的绑架，没有云盘的门禁，没有订阅的弹窗，只有一块素净的编辑区和一块忠实的预览区。

这就是 mdviewer 最初的样子。一个 HTML 文件、一个 CSS 文件、一个 JavaScript 文件，加上一个 Service Worker 让它能在离线时照常工作。它小到可以塞进任何角落，又稳到可以长期依赖。

但"朴素"从来不是终点。当一支笔用顺了手，你自然会想要一个笔筒、一个抽屉、一个书架——把写过的东西收好，下次随手就能取用。我们的协作，正是沿着这条"从笔到书房"的线索，一点一点长起来的。

这篇文章想记录的，不只是最终产品的功能清单，更是那些在对话中反复推敲、在取舍间来回掂量的瞬间：**我们如何决定一个功能该长成什么样子，又如何在一遍遍打磨里把"能用"变成"好用"。**

---

## 二、地基：克制的技术栈

在谈功能之前，有必要先说说它的骨架，因为骨架决定了它能长多大、长多稳。

mdviewer 几乎没有依赖。没有 React，没有打包器，没有前端框架。整个应用是原生 JavaScript 写就，直接跑在浏览器里。这听起来像是一种"复古"，但恰恰相反——这是一种清醒的克制。

**为什么克制反而是一种力量？**

第一，零构建意味着零等待。改一行 CSS，刷新就能看到；改一个函数，立刻生效。我们后来做了那么多轮快速迭代，能在一天之内来回几十次微调，靠的正是没有编译链在中间拦着。

第二，IndexedDB 是真正的本地数据库。很多人以为"网页应用"等于"数据在云端"，mdviewer 偏不。我们的文库把每一篇文档存在浏览器自带的 IndexedDB 里，按域名隔离，不依赖任何服务器。这意味着：你断网了，文档还在；你换电脑（同一浏览器 profile）登录，文档还在；没有人能拿去训练模型，因为数据压根没出过你的机器。

第三，Service Worker 让它像 App。配合 manifest，mdviewer 可以被"安装"到桌面或手机主屏，离线打开，体验接近原生。这也是为什么我们后来能放心地往里加"文库""音乐"这些重度功能——底子是稳的。

这种技术选择背后是一种信念：**工具应当服务于人，而不是反过来。简单，是可维护性的前提；本地优先，是隐私的底线。**

---

## 三、转折：文库的出现

真正的转折发生在我们决定"给这支笔配一个书架"的那一刻。

最初的需求只是一个模糊的念头：文档库没有实现完整，请继续。没有详细规格，没有交互稿，只有一个方向——让写过的东西能被收起来、再被找出来。

我们做的第一个关键决策，是**形态**。

我给了你两个选项：文档库面板用哪种形式？你选了"左侧抽屉"。这个选择看似随意，实则影响深远。抽屉意味着"不打断"——它从左侧滑出，覆盖在主界面之上，看完一推就走，不占编辑区的常驻空间。相比把文库做成常驻的侧边栏，抽屉更轻，更符合一个"编辑器"的主体定位：编辑永远是第一顺位，文库是随时召之即来的副手。

第二个关键决策，是**保存方式**。

打开文库里的文档后，编辑的内容怎么保存？我建议了"自动回写"。你认可了。这决定了文库不是一个"导入导出"的搬运工，而是一个**活的书架**——你在这里打开一篇文章，随手改几个字，它自己就记下了，不需要你像在传统编辑器里那样反复按 Ctrl+S、找保存路径、担心覆盖了原文件。

这两个决策合在一起，定义了文库的灵魂：**它不是一个文件管理器，而是一块有记忆的玻璃。你写，它记；你走，它守。**

为了把这个灵魂落到实处，我们在 app.js 里接了一条"去抖自动回写"的链路：每次编辑停 800 毫秒，就把当前文档的内容、文件名、更新时间写回 IndexedDB，状态栏同步亮起一个"✓ 已存文库"的小提示。用户几乎感知不到保存这件事在发生——而这恰恰是好的保存设计的最高境界。

## 四、打磨：那些被认真对待的细节

功能能跑通只是第一步。真正让一个项目"完善"的，是那些容易被忽略、却真实影响手感的细节。我们在这上面花的时间，可能比写主功能还多。

**副标题的由来。** 你提出，文库打开时左上角主标题"文库"右侧，可以加一行小一号的副标题。我们选了"苑广山的在线文库"。这不只是装饰——它把"一个工具"轻轻变成"一个人的书房"，给了产品一种归属感和温度。样式上我们用 `var(--text-dim)` 把它压成次要层级，不抢主标题的视觉权重，明暗主题下都自然。

**一个被修掉的隐形 bug。** 你很敏锐地发现：点击文库里的条目时，更新时间也跟着变了，这不对——时间应该只在真正修改保存时才更新。我排查后确认，根因是打开文档时调用了 `afterChange()`，而它的末尾会触发回写、顺手刷新时间戳。修法是给 `afterChange` 加了一个 `skipWriteback` 开关，让"打开"只渲染、不写时间，回写严格限定在真正的编辑输入上。我们为此写了专门的测试用例，确认"打开不改时间、编辑才改时间"，4 项断言全过。

这件事值得单独记一笔，因为它体现了一整套工作方法：**用户的体感永远比代码逻辑更可靠。** 代码"能跑"，但用户的直觉告诉我们"哪里怪怪的"。顺着那点怪，往往能挖出真正的瑕疵。

**署名。** 你在菜单底部加了一句"Designed by yuanguangshan."。它小到几乎不占地方，却是一个作品应有的签名。我们把它做成不可点击的展示项，小号、居中、弱化，恰到好处地收尾整个菜单。

## 五、扩展：文库的三次生长

文库站稳之后，你又陆续给它加了三样东西，每一次都让它更像一个完整的"个人知识空间"。

**第一次：添加到文档库。** 你从电脑打开了一篇文章，想把它收进文库。于是我们在"⋯"菜单里加了"📥 添加到文档库"——一键把当前编辑器里的内容（连同文件名）转存进文库，并设为当前文档，从此它的编辑也自动回写。这一下打通了"外部文章"与"本地书房"的任督二脉：你不再是只能在文库里新建，而是能把任何来源的文字收编进来。

**第二次：上传。** 抽屉里加了"⬆️ 上传"按钮，背后是一个隐藏的多选文件选择器。电脑上、手机上都能用，可以一次选好几篇本地已有文档，逐个读入文库。这补齐了"我有现成文件"的场景，让文库从一个"边写边存"的地方，变成一个"集中托管"的地方。

**第三次：下载。** 每个文库条目除了"重命名"和"删除"，又多了一个"⬇️ 下载"，把单篇内容导出成 .md 文件。操作和编辑、删除并排，逻辑对称、触手可及。至此，文库在"存入—管理—取出"这条链路上形成了闭环：**添加到文库 / 上传 是进，下载 / 导出是出，自动回写是活的血液。**

三次生长，方向都很清楚：让文库既能"收得进"，也能"拿得出"。它没有把自己锁死在单一流向里。

## 六、美学：让界面自己会呼吸

做到这里，功能已经相当齐备。但你提了一个更"软"、却更见功力的问题：还有哪些界面元素可以优化得更好看一点？

这拉开了我们对视觉语言的系统梳理。我没有笼统地"美化"，而是按性价比排了序，挑出改动小、收效大的几处，逐一落地：

- **工具栏的层次。** 原来一排按钮视觉权重相同，略显平板。我们把"文库 / 新建 / 打开 / 更多"改成幽灵按钮——透明底，只有 hover 时才微微显出底色；只留"保存"一个实心主色按钮。主次的分别一旦拉开，界面立刻有了焦点。
- **文库条目的头像。** 每篇文档左侧加了一个首字母色块头像，当前项再加一条主色竖线。原本两行文字的"列表感"，变成了有身份、有选中态的"书脊感"。
- **主题化细滚动条。** 系统默认滚动条在深色背景下格外扎眼。我们换成跟随主题的细滚动条，编辑区、预览区、文库、菜单都统一了语言。
- **菜单的入场动效。** 弹出加了 90 毫秒的淡入加轻微上移，不再是硬邦邦地出现。
- **移动端的防溢出。** 我们额外加了一条 `≤400px` 的断点，收紧间距、隐藏计数角标、缩小 logo，确保超窄屏手机也不挤。

这里我特别想记下你对一个设计取舍的追问：**视图切换按钮会不会在手机上太挤？** 我核查后发现，它本来就是"分屏 / 编辑 / 预览"单按钮循环切换，只占一格，根本不会铺开——所以你担心的"三个并排挤成一团"从一开始就不会发生。但我没有含糊带过，而是把这个事实讲清楚，并明确"绝不做成三段按钮"作为约束。这件事的启示是：**好的协作不是一味照单执行，而是帮你确认担心是否真的成立，并给出有依据的判断。**

## 七、收尾：装饰与温度

功能与美学之外，你还关心"空"的地方。

文库抽屉下方原本有点空。我们加了一条常驻底部栏：左侧显示"共 N 篇"（数字随文档数实时更新），右侧一句"🔒 本地保存 · 不上传"。它既填了留白，又顺手给用户一个安心承诺——数据不出本机。空状态也不再是干巴巴一行字，而是配了一幅柔和的 SVG 文档插画，下附引导文案。

最后，你提议在文库里内嵌一个音乐播放器，网址是 `https://wealth.want.biz/music/index.html`。我们把它做成一个可折叠的 🎵 面板：默认收起，点开才加载内嵌 iframe，不挤占文档列表。展开后你看到了音乐列表，又顺手说"高度翻倍"——我们把面板从 210px 调到 420px，让播放列表看得更舒服。这处小小的音乐角落，让一个写作工具多了一点生活的余韵：**写累了，推开通栏，听听歌。**

我尤其想记下这里的一个工程纪律：内嵌第三方网页有被 `X-Frame-Options` 拦截的风险。我探了目标站点的响应头，确认没有反嵌策略后才放心接入，并额外保留了一个"新标签打开"的兜底按钮。这是"做得漂亮"和"做得稳妥"之间的平衡——好看，但不能是脆弱的好看。

## 八、质量：我们如何相信它没坏

功能越加越多，最怕的是"改一个、崩一片"。我们建立了一套轻量却有效的质量保障：用 jsdom 搭一个无头 DOM 环境，配 fake-indexeddb 模拟浏览器数据库，把文库的关键流程写成可重复运行的断言。

在这些测试里，我们验证过：

- 新建文档能正确写进 IndexedDB；
- 编辑后自动回写的内容、文件名都正确；
- 重命名同步生效；
- 删除后数据库清空，且当前文档的旧草稿被一并清除（我们真在这里抓出过一个 bug：删除当前文档后旧草稿会残留，重载时"复活"）；
- 刷新页面能自动恢复上次的文档；
- 打开文档不刷新时间戳、只有编辑才刷新。

每一轮较大的改动，我都会跑一遍这些测试再交付。它们像一张安全网，让我们敢于大胆迭代——因为一旦哪里断了，测试会立刻叫停。**这是一种信任的建立方式：不是嘴上说"我测过了"，而是把验证变成可复现的证据。**

## 九、协作：人机共创的另一种可能

写到这儿，文章其实已经超出"产品评述"的范畴，更像一篇"协作观察"。

我们之间的关系，不是"你下命令、我执行"的流水線，而更接近**两个人对着一块白板来回推演**。你给出方向（"文档库用左侧抽屉""编辑自动回写""加个音乐播放器"），我在方向上补全细节、预判坑点、给出可选方案；你再基于我的反馈做决定，或提出新的直觉（"点击时时间也变了，这不对吧"）。

这种来回里最珍贵的，是**你始终在把关"手感"**。你不一定写代码，但你比谁都清楚"用起来舒不舒服"。而我的角色，是把手感翻译成实现，并在翻译过程中把那些你未必会注意的边界情况（时间戳、旧草稿复活、iframe 拦截）提前兜住。

它印证了一件事：当人负责"想要什么、什么感觉对"，机器负责"怎么稳稳地实现、并守住边角"，协作的产出会明显好过任何一边单独硬扛。

## 十、回望与未尽

回头看，mdviewer 从一个朴素的双栏编辑器，长成了一个有文库、有导出、有美学、有音乐的小小书房。版本号从 v1.10.6 一路走到 v1.11.5，每一次提交背后都是一次具体的对话。

当然，它仍有可以生长的地方，留作未来的线索：

- 文库目前按浏览器域名隔离，换设备不互通。若要跨设备同步，需要一条谨慎的同步通道，且必须先把隐私边界讲清楚再动。
- 可以支持"导入本地 .md 进文库"与"单篇导出为文件"的更细粒度操作。
- 音乐播放器可支持自定义歌单网址，而不只内嵌单一站点。
- 文库文档之间的双向链接、标签、全文搜索，会把它从"书架"推向"知识网络"。

但这些都不是缺憾，而是下一程的路标。一个项目"完善"，不等于"终结"——它只是到了一个值得停下来、写一篇文章纪念的节点。

---

## 十一、结语

我们常以为，好的工具是设计出来的。但 mdviewer 告诉我：**好的工具，是一起聊出来的。**

你的一句话——"文档库用左侧抽屉""编辑自动回写""高度翻倍""加个署名"——看似轻巧，却一次次把方向拨正。我的价值，不在于写出了多少行代码，而在于把你的直觉稳稳地接住、落地、并守好那些你看不见的接缝。

所以这篇文章的标题叫"从一支笔到一座书房"。笔是你给的初衷，书房是我们一起砌起来的。它或许永远不会出现在谁的年度榜单上，但它真实地、安静地，陪你写了那么多字、存了那么多篇、放了那么一首歌。

这就够了。

*——写于我们共同把版本推到 v1.11.5 的那天。Designed by yuanguangshan，也，与我们一起。*
